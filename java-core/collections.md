### ArrayList vs LinkedList

ArrayList
- ✅ быстрый доступ по индексу
- ✅ лучше использует CPU cache
- ✅ меньше памяти
- ✅ быстрее в большинстве случаев
- ❌ медленные вставки в середину
- ❌ resize массива (копирование)

LinkedList
- ✅ быстрое добавление в начало
- ✅ быстрое удаление узла если есть ссылка
- ✅ удобно использовать как Deque
- ❌ медленный доступ по индексу
- ❌ больше памяти
- ❌ плохая cache locality
- ❌ больше нагрузка на GC

P.S.: Cache locality (локальность кэша)
Это свойство алгоритма или структуры данных эффективно использовать CPU cache, обращаясь к данным, которые лежат рядом в памяти.

| Критерий                   | ArrayList                  | LinkedList                  |
| -------------------------- | -------------------------- | --------------------------- |
| Структура                  | Динамический массив        | Двусвязный список           |
| Доступ по индексу `get(i)` | **O(1)**                   | **O(n)**                    |
| Добавление в конец         | **O(1)** (амортизированно) | **O(1)**                    |
| Добавление в середину      | **O(n)** (сдвиг элементов) | **O(n)** (поиск позиции)    |
| Удаление из середины       | **O(n)**                   | **O(n)**                    |
| Добавление в начало        | **O(n)**                   | **O(1)**                    |
| Использование памяти       | меньше                     | больше (prev + next ссылки) |
| Cache locality             | хорошая                    | плохая                      |
| Итерация                   | быстрее                    | медленнее                   |

### Алгоритмы поиска в стандартных коллекциях Java / Kotlin

В стандартных коллекциях используются три основных подхода:

Линейный поиск — в списках ArrayList/LinkedList O(n)
Хеширование — в HashMap/HashSet O(1) в среднем
Поиск в дереве — в TreeMap/TreeSet O(log n)

Также есть бинарный поиск через Collections.binarySearch для отсортированных данных.

| Коллекция                    | Алгоритм поиска                     | Сложность             | Суть                                     |
| ---------------------------- | ----------------------------------- | --------------------- | ---------------------------------------- |
| `ArrayList`                  | Линейный поиск                      | O(n)                  | Последовательно проверяет элементы       |
| `LinkedList`                 | Линейный поиск                      | O(n)                  | Идёт по узлам списка                     |
| `HashMap`                    | Хеширование + поиск в бакете        | O(1) avg / O(n) worst | По hash → бакет → поиск внутри           |
| `HashSet`                    | Хеширование                         | O(1) avg              | То же, что HashMap (ключи)               |
| `LinkedHashMap`              | Хеширование                         | O(1) avg              | + сохранение порядка                     |
| `TreeMap`                    | Поиск в дереве (BST, красно-чёрное) | O(log n)              | Бинарный поиск по дереву                 |
| `TreeSet`                    | Поиск в дереве                      | O(log n)              | Основан на TreeMap                       |
| `PriorityQueue`              | Поиск НЕ оптимизирован              | O(n)                  | Только heap (быстрое извлечение min/max) |
| `Collections.binarySearch()` | Бинарный поиск                      | O(log n)              | Для отсортированных списков              |

| Алгоритм                           | Суть работы                                                                                              | Сложность (в среднем / худшая)       | Требуется сортировка |
| ---------------------------------- | -------------------------------------------------------------------------------------------------------- | ------------------------------------ | -------------------- |
| **Linear Search (Линейный поиск)** | Последовательно проверяем каждый элемент массива, пока не найдём нужный                                  | O(n) / O(n)                          | ❌ Нет                |
| **Binary Search (Бинарный поиск)** | Делим отсортированный массив пополам и отбрасываем половину элементов на каждом шаге                     | O(log n) / O(log n)                  | ✅ Да                 |
| **Jump Search**                    | Прыгаем по массиву блоками (обычно √n), чтобы найти диапазон, затем выполняем линейный поиск внутри него | O(√n) / O(√n)                        | ✅ Да                 |
| **Interpolation Search**           | Предсказываем позицию элемента на основе его значения (если данные распределены равномерно)              | O(log log n) / O(n)                  | ✅ Да                 |
| **Exponential Search**             | Быстро находим диапазон, увеличивая индекс экспоненциально (1,2,4,8…), затем применяем бинарный поиск    | O(log n) / O(log n)                  | ✅ Да                 |
| **Fibonacci Search**               | Делит массив на части с использованием чисел Фибоначчи вместо середины                                   | O(log n) / O(log n)                  | ✅ Да                 |
| **Depth First Search (DFS)**       | Идём по графу максимально глубоко по одной ветке, затем возвращаемся назад                               | O(V + E)                             | ❌ Нет                |
| **Breadth First Search (BFS)**     | Обходим граф по уровням, сначала всех соседей текущей вершины                                            | O(V + E)                             | ❌ Нет                |
| **A* Search**                      | Находит кратчайший путь, используя стоимость пути + эвристическую оценку до цели                         | O(E) примерно (зависит от эвристики) | ❌ Нет                |

### Линейный поиск (Linear Search)

⏱ O(n) / O(n) / O(n)
💾 O(1)

🧠 Как работает

Проходим массив последовательно и сравниваем каждый элемент с искомым.

⚠️ Особенности
- Работает на неотсортированных данных
- Очень простой
- Плохо масштабируется

### Бинарный поиск (Binary Search)

⏱ O(1) / O(log n) / O(log n)
💾 O(1) (итеративный), O(log n) (рекурсивный из-за стека)

🧠 Как работает
- Требует отсортированный массив
- Делим диапазон пополам
- Сравниваем с серединой
- Отбрасываем половину

Каждая итерация делит область поиска на 2 → логарифмическая сложность.

⚠️ Частые вопросы
overflow при `mid = (l + r) / 2`
Правильно: `mid = l + (r - l) / 2`
lower_bound / upper_bound

### HashTable
HashSet<E> хранит только ключи, а значения в HashMap не нужны. 
Поэтому внутри HashSet создаётся приватный HashMap<E, Object>

```java
public class HashSet<E> extends AbstractSet<E> implements Set<E>, Cloneable, java.io.Serializable {
    private transient HashMap<E,Object> map;

    // Просто фиктивное значение для всех ключей
    private static final Object PRESENT = new Object();
    
    public HashSet() {
        map = new HashMap<>();
    }

    public boolean add(E e) {
        return map.put(e, PRESENT) == null;
    }
}
```

Каждый bucket — связанный список (LinkedList) узлов
HashMap с Java 8+:
- Если в bucket слишком много элементов (больше 8), список преобразуется в красно-черное дерево → O(log n) вместо O(n) для длинного списка
- Это увеличивает производительность при коллизиях
- Если после resize количество элементов в бакете < 6 → tree → обратно linked list

loadFactor — это коэффициент заполнения HashMap, который показывает, 
насколько заполнен массив бакетов до того, как HashMap будет увеличен.

Формула: `Threshold`=`capacity`×`loadFactor`
- capacity — текущее количество бакетов (длина массива table)
- loadFactor — обычно 0.75 (по умолчанию)
- threshold — количество элементов, при достижении которого HashMap будет расширен

Поиск бакета:
- Раньше: остаток деления hashcode на capacity `hash % capacity`
- C Java 8+:   
```java
int index = (n - 1) & hash;
```
- n — текущая capacity (длина массива table)
- hash — хеш-код ключа (обычно после hash ^= (hash >>> 16) для равномерного распределения)
- & — побитовое AND

При добавлении новой записи в HashSet 
- считаем хешкод объекта
- на основе хеша и размера сета находим индекс бакета ()
- записываем в нужный бакет объект

Если при добавлении новой записи мы получаем уже занятый индекс бакета,
хотя объекты не равны, то мы получаем коллизию

В таком случае мы добавляем в список, хранимый в бакете новый элемент

Как работает `contains` или `remove`
- считаем хешкод объекта
- находим индекс бакета
- если элемент один то вернем/удалим его
- если элементы бакета не содержат наш объект то значит его нет в коллекции. сравниваем мы через искомое через equals

По сути операции добавления, удаления и проверки на наличие
используют поиск в бакете
Мы стремимся к тому, что бы в одном бакете был только один элемент

Если бы хешсет не увеличивался, то в бакетах накапливалось много объектов
и поиск внутри бакета порушил нашу скорость операций

По этому мы вынуждены наращивать наш ассоциативный список.
При добавлении нового элемент мы:
- проверим нет ли уже этого элемента через `contains`
- проверим не нужно ли выполнить наращивание на базе `loadFactor`
  - создаем новый "массив" на x2 бакетов
  - вставляем все элементы в новый массив с новым размером
- добавляем новый объект

По этому при добавлении у нас может быть сложность O(n) при приращении массива
- Resize дорогой по CPU, так как происходит перераспределение всех элементов
- Значение loadFactor 0.75 — компромисс между памятью и скоростью поиска
  - Меньший loadFactor → больше памяти, меньше коллизий
  - Больший loadFactor → меньше памяти, больше коллизий

HashMap работает аналогично.
Хешкод для поиска бакета считается по ключу, записываем мы не объект, а `Map.Entry<K,V>`.

Трюки:
Если `hashCode()` будет возвращать одно и то же значение, 
то все элементы попадут в один бакет и сложность поиска будет O(n)

Если `hashCode()` будет возвращать рандомное число,
то при поиске элемента мы будет искать в ложном бакете и поиск будет сломан

по этому вызов `hashCode()` должен быть идемпотентным если данные не менялись

Если у нас неканоничный `equals()`, то и `hashcode()` должен работать по той же логике.
Что бы для равных по `equals()` объекты у них был тот же `hashcode()`.

HashMap и отрицательные hashCode: hashCode может быть отрицательным, используется битовая маска hash & (n-1) → всегда в диапазоне \[0, capacity-1]

Порядок элементов не гарантирован (LinkedHashSet нужен для сохранения порядка)

HashSet не потокобезопасен (ConcurrentHashMap.newKeySet() для потоков).
Resize может вызвать infinite loop, если несколько потоков одновременно модифицируют HashMap.

Если заранее известно количество элементов n, 
лучше создать HashMap(n / 0.75 + 1, 0.75)
чтобы избежать многократного resize
