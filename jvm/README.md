# Java Virtual Machine

### GC
Сборщик мусора (Garbage Collector, GC) в Java определяет, какие объекты можно удалить, 
используя алгоритмы обхода графа объектов и учёта ссылок.
Основные подходы — это достижимость (reachability) и алгоритмы маркировки.

Объект считается достижимым, если на него есть цепочка ссылок из корневых точек (GC Roots). 
Если такой цепочки нет — объект недостижим и может быть удалён.

##### GC Roots (Корневые точки):
- Локальные переменные в стеке (методы, вызовы функций).
- Статические поля классов.
- Активные потоки (Thread).
- Ссылки в JNI (нативные методы).
- Загруженные системные классы (например, Class-объекты).

##### Алгоритмы поиска мусора

a) Mark and Sweep

Маркировка (Mark):
- GC обходит все объекты, начиная с GC Roots, и помечает достижимые.
- Используется обход в глубину (DFS) или в ширину (BFS).

Очистка (Sweep):
- Память сканируется, и все непомеченные объекты удаляются.
- Память не уплотняется (может привести к фрагментации).

b) Generational GC
- Основан на гипотезе: большинство объектов умирают молодыми.
- Память делится на молодое и старое поколения.

Молодое поколение:
Сборка мусора здесь происходит часто и очень быстро

Старое поколение:
Туда перемещаются объекты, которые пережили несколько сборок мусора в Young Generation. 
Сборка мусора здесь происходит реже.

c) G1 Garbage Collector (Garbage-First)
- Делит кучу на регионы (regions), собирает мусор в приоритетных регионах.
- Работает параллельно и инкрементально (минимизирует паузы).

### Stack
Хранит примитивы и ссылки на объекты в рамках выполнения методов.

Работает по LIFO (Last In, First Out).

Быстрый доступ, но фиксированный размер (может привести к StackOverflowError при глубокой рекурсии).

Каждый поток имеет свой стек.

### Heap
Хранит объекты (экземпляры классов, массивы).

Динамически расширяется (но ограничен размером памяти устройства).

Общий для всех потоков, требует синхронизации.

Подвержен фрагментации (но в ART это решено лучше, чем в Dalvik).
