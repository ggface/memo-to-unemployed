### REST
REST — это архитектурный стиль для построения сетевых API. Расшифровывается как Representational State Transfer.

1) Все в REST — это ресурсы. Ресурс — это просто сущность, с которой ты работаешь.

Примеры:
```
/users
/users/1
/posts/42/comments
```

2) HTTP-методы = действия

| Метод  | Что делает         | Пример          |
| ------ | ------------------ | --------------- |
| GET    | получить данные    | GET /users      |
| POST   | создать            | POST /users     |
| PUT    | полностью обновить | PUT /users/1    |
| PATCH  | частично обновить  | PATCH /users/1  |
| DELETE | удалить            | DELETE /users/1 |

3) Статусы ответа (HTTP Status Codes)

| Код | Значение     |
| --- | ------------ |
| 200 | OK           |
| 201 | Created      |
| 204 | No Content   |
| 400 | Bad Request  |
| 401 | Unauthorized |
| 403 | Forbidden    |
| 404 | Not Found    |
| 500 | Server Error |

4) Stateless (без состояния)
   Каждый запрос самодостаточный.

❗ Сервер не хранит состояние клиента между запросами.

Это значит что токен нужно передавать каждый раз.
```http request
GET /profile
Authorization: Bearer token
```

### GraphQL
GraphQL — это язык запросов, который позволяет клиенту запрашивать только нужные данные.
В Android обычно используется через Apollo, который генерирует типобезопасные модели и поддерживает кеширование.

Плюсы:
- Нет overfetching (Получение лишних данных, которые не используются)
- Нет underfetching (Нужно делать несколько запросов, чтобы собрать данные)
- Типобезопасность
- Один endpoint
- Удобно для сложных экранов - один запрос → сразу всё

Минусы:
- Кэш сложнее
- Сложнее дебажить
- Большие ответы при плохих запросах

🔷 Основные концепции GraphQL
1. Schema - описание API
```
type User {
   id: ID!
   name: String!
}
```
2. Query - получение данных
```
query {
   user(id: "1") {
      name
   }
}
```
3. Mutation - изменение данных

```
mutation {
    createUser(name: "John") {
      id
    }
}
```
4. Subscription - realtime обновления (WebSocket)

### Когда лучше GraphQL, а когда REST?
GraphQL:
- Сложные экраны с агрегацией данных
- Ограниченный мобильный интернет (меньше запросов, меньше payload)
- Быстро меняющийся UI (для рест нужен новый запрос на беке)

REST:
- Простые CRUD API (если API простой → REST проще и дешевле)
- Кэширование (если важен кэш на уровне HTTP/CDN → REST лучше)
- Большие файлы (upload/download)
- Контроль нагрузки (endpoints контролируются сервером, а в GraphQL клиент может случайно положить сервер)

### CRUD

CRUD — это базовый набор операций для работы с данными.

Расшифровка:
- C — Create (создать)
- R — Read (прочитать)
- U — Update (обновить)
- D — Delete (удалить)
