# Пагинация

Лучший вариант это Cursor-based pagination
```http request
GET /items?cursor=4651
```

Значение курсора обычно id последнего загруженного элемента или timestamp.
- не ломается при добавлении новых элементов
- идеальна для real-time лент
- без сдвига страниц
- отлично работает c бесконечной прокруткой
- хорошо в мобильных сетях

❌ Offset pagination (page = 1,2,3…)
```http request
GET /items?page=2
```
- при добавлении новых элементов данные съезжают
- непредсказуемость
- race conditions
- не подходит для real-time

❌ Limit/offset
```http request
GET /items?limit=20&offset=40
```
- те же проблемы: вставка в середину ломает данные
- сложно поддерживать консистентность

### Пагинация для изменяемых списков

💡 Как решать
- загружаешь remote data
- сохраняешь в таблицу products (immutable)
- локальные изменения — в user_updates
- UI подписан на JOIN таблиц
- результат = merge(remote + local changes)

✔ remote данные могут обновляться пагинацией

✔ локальные изменения не исчезают

✔ UI всегда получает актуальное состояние

✔ архитектура не требует full reload

✔ хорошая масштабируемость
